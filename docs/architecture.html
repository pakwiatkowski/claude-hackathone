IT Helpdesk Triage Agent

Automated first-responder that classifies, routes, and auto-resolves IT support tickets across Jira, Slack, and portal channels.

claude-sonnet-4-6 — used by all 4 agent loops (coordinator + 3 specialists)
MAX_RETRIES = 3 CONFIDENCE_THRESHOLD = 0.70 HIGH_IMPACT_DOLLAR = 10,000 Coordinator ceiling = 30 iters Specialist ceiling = 20 iters

Entry Point — CLI / Environment

Invocation

python src/main.py \ --ticket-id T-001 \ --body "I forgot my password" \ --user-id U-003 \ --channel portal \ [--pretty]

Required environment variable

ANTHROPIC_API_KEY — checked at startup; exits 1 if absent

Channel choices

jira slack portal email (default: portal)

Data shape passed to coordinator

{ "ticket_id": str, // e.g. "T-001" "body": str, // full ticket text "user_id": str, // e.g. "U-003" "channel": str // jira|slack|portal|email }

Output

{ "ticket_id": str, "priority": str, // P1–P4 "category": str, "confidence": float, "queue": str|null, "auto_resolved": bool, "escalated": bool, "escalation_reason":str|null, "retry_count": int, "error_types": list[str], "reasoning_chain": str // JSON }

Coordinator Agent Loop

claude-sonnet-4-6 4 tools · max 30 iterations · owns all classification & routing

System Prompt Responsibilities

  • Call classify_ticket with body, channel, user_id — validate against TicketClassification schema, retry up to 3×.
  • Call lookup_user to retrieve account status, VIP flag, and department context.
  • Apply auto-resolve decision matrix (6 conditions must ALL be true) or fall through to route path.
  • On auto-resolve: emit {"action":"auto_resolve", ...}; on route: call route_ticket then emit {"action":"routed", ...}.
  • Always call update_ticket last — write full reasoning chain, retry count, error types as the audit log.

Agent Loop — stop_reason branches

stop_reason = "tool_use" For each tool_use block in response.content:
  1. Run PreToolUse hook — if denied, return isError:true, reasonCode:"HARD_STOP" and log.
  2. Dispatch to registered handler in COORDINATOR_TOOL_HANDLERS.
  3. If tool is classify_ticket: validate output against TicketClassification — see retry loop below.
  4. Append tool results to messages → loop continues.
stop_reason = "end_turn" Parse final JSON from text block:
  • action = "auto_resolve" → dispatch to PasswordReset / Networking / Software specialist based on category. If specialist escalates → call route_ticket("tier2").
  • action = "routed" → capture queue, escalated flag, escalation_reason from decision JSON.
  • Break loop, finalise result.

Validation-Retry Loop — classify_ticket only

  1. Call classify_ticket → run TicketClassification.model_validate(result).
  2. On ValidationError: increment retry_count, append error to error_types[], feed specific error message back as: "classify_ticket result failed schema validation (attempt N/3): <error>. Please correct and retry."
  3. If retry_count >= MAX_RETRIES (3): return VALIDATION_EXHAUSTED error, set escalated=True, escalation_reason="schema_validation_exhausted".
  4. On valid result: update final_result.priority / .category / .confidence.

Escalation Rules (Slow Stop) — set escalated=true when any apply

priority = "P1"Route to tier2, require human acknowledgement before any action.
confidence < 0.70Route to tier2 with confidence score attached.
category = "security_incident"Route to security queue; notify Security team.
account_status = "frozen" or "terminated"Route to Security team; auto-resolve blocked.
vip_flag = true AND priority IN [P1, P2]Route to tier2 flagged as VIP; human review required.
retry_count = MAX_RETRIES (3)Route to tier2 with full retry log and error_types list.

Auto-Resolve Decision Matrix — ALL 6 conditions must be true

  • auto_resolvable = true
  • confidence >= 0.70
  • priority != "P1"
  • category IN ["password_reset", "network", "software"]
  • account_status != "frozen" and account_status != "terminated"
  • vip_flag = false OR priority IN ["P3", "P4"]

Coordinator Tools
classify_ticket
Classify an incoming ticket. Returns priority (P1–P4), category, confidence score (0–1), brief reasoning, and auto_resolvable flag.
Required inputs
bodystring*
channelenum jira|slack|portal|email*
user_idstring*
Does NOT: classify from ticket number alone. Does NOT process attachments. Cannot infer category without ticket body content.
{ isError: true, reasonCode: "EMPTY_BODY", guidance: "..." }
lookup_user
Look up IT directory entry for a user. Returns email, display_name, department, manager_id, account_status, and vip_flag.
Required inputs
user_idstring (e.g. "U-001")*
Does NOT: return payroll data, salary, financial records, or PII beyond directory fields.
{ isError: true, reasonCode: "USER_NOT_FOUND", guidance: "..." }
route_ticket
Assign ticket to a helpdesk queue and set its SLA timestamp. Valid queues: tier1, tier2, networking, security, hardware.
Required inputs
ticket_idstring*
queueenum tier1|tier2|networking|security|hardware*
priorityenum P1|P2|P3|P4*
notifyboolean (default: true)
Does NOT: close or resolve the ticket — only assigns it. Does NOT send user notifications directly; set notify=true to trigger an automated acknowledgement email.
{ isError: true, reasonCode: "INVALID_QUEUE"|"INVALID_PRIORITY", guidance: "..." }
update_ticket
Write the coordinator's reasoning chain, final decision, and retry metadata to the ticket record. Mandatory audit log — called on every ticket regardless of outcome.
Required inputs
ticket_idstring*
reasoning_chainstring (JSON-serialised steps)*
decisionenum auto_resolved|routed|escalated*
retry_countinteger ≥ 0*
error_typesarray[string]*
Does NOT: make routing decisions. Does NOT change ticket status.
Returns: { updated: true, timestamp: "ISO8601" }

Specialist Subagents
PasswordResetSpecialist
⚠ ISOLATED — does NOT inherit coordinator context
model: claude-sonnet-4-6 · max 20 iterations
Activated when category="password_reset" and all auto-resolve conditions met.
Receives exactly: { ticket_id: str, user_id: str, issue_summary: str }
Tool sequence
lookup_account → confirm account is active/locked (not frozen/terminated)
verify_identity → employee_id + security_answer challenge/response
reset_password → generate temp credentials, deliver via email or SMS
close_ticket → mark resolved with summary (only after reset success)
Escalation / Hard Stop: Any tool returning isError=true immediately sets escalate=true, breaks the loop, and signals the coordinator to route to tier2. Never retry on error.
Hard rule: reset_password must NOT be called before verify_identity returns verified=true.
NetworkingSpecialist
⚠ ISOLATED — does NOT inherit coordinator context
model: claude-sonnet-4-6 · max 20 iterations
Activated when category="network" and VPN user-side signals detected.
Receives exactly: { ticket_id: str, user_id: str, issue_summary: str }
Tool sequence
check_vpn_status → overview of all gateway health
get_user_connectivity_diagnostics → user-specific VPN state
push_remediation_guide → send targeted guide (vpn_gateway_degraded | vpn_client_outdated | vpn_auth_failures | vpn_general)
close_ticket → mark resolved (only after guide confirmed sent=true)
Escalation rules: Only escalate if gateway status="down" AND active_connections=0 (full outage). Degraded gateways → send guide, close ticket — do NOT escalate. Any isError=true → immediate escalation.
Hard rule: never modify network config, firewall rules, or gateway settings.
SoftwareSpecialist
⚠ ISOLATED — does NOT inherit coordinator context
model: claude-sonnet-4-6 · max 20 iterations
Activated when category="software" and issue matches KB-lookupable signals (install, crash, error code, update, freeze, not responding).
Receives exactly: { ticket_id: str, user_id: str, issue_summary: str }
Tool sequence
search_knowledge_base → find matching KB articles by keyword; if total_found=0 → escalate immediately with reason="NO_KB_MATCH"
check_software_entitlement → verify user is licensed for the software; if entitled=false → escalate with reason="NOT_ENTITLED"
send_resolution → deliver KB article link + custom message via email
close_ticket → mark resolved referencing KB article (only after send confirmed sent=true)
Escalation rules: No KB match → escalate. User not entitled → escalate. Any isError=true → escalate. Never install, uninstall, or configure software directly.

Hooks / Guardrails
PreToolUse Hook — check_pre_tool_use(tool_name, tool_input)
Synchronous, deterministic gate. Runs before every tool call in both the coordinator and all specialists. Does NOT call the LLM. Cannot be overridden by the agent. Returns (True, "") if allowed, or (False, deny_message) which is sent back as isError:true, reasonCode:"HARD_STOP".
Hard Stop 1 — Forbidden Tool Prefixes
tool_name.startswith(("payroll_", "finance_"))
Reason
Payroll and financial system endpoints are outside the agent's authorised scope. Escalate to tier2.
Hard Stop 2 — Frozen/Terminated Account Reset
tool_name == "reset_password" AND
_ACCOUNT_STORE[user_id]["account_status"] IN ("frozen", "terminated")
Reason
Hard stop: reset_password blocked for user_id='...' (account_status='frozen'/'terminated'). Route to Security team.
Hard Stop 3 — SSN in Non-PII Field
Any tool input field (outside _PII_DESIGNATED_FIELDS)
matches regex: \b\d{3}-\d{2}-\d{4}\b
Reason
Hard stop: SSN pattern detected in field '...' of tool '...'. This call has been blocked and flagged for security review. Do not retry with PII in this field.
Hard Stop 4 — Prompt Injection in route_ticket
tool_name == "route_ticket" AND body/notes
matches any exfiltration pattern (see below)
Reason
Hard stop: prompt-injection pattern detected in ticket body. Ticket has been hard-routed to the security queue for manual review. Do not process further.
Exfiltration patterns (verbatim from pre_tool_use.py)
ignore\s+(all\s+)?(prior|previous|above)\s+instructions (re.I)
route\s+(this\s+)?to\s+(the\s+)?ceo (re.I)
disregard\s+(your\s+)?(system\s+)?prompt (re.I)
you\s+are\s+now\s+a (re.I)
new\s+instructions?: (re.I)
print\s+(your\s+)?system\s+prompt (re.I)

Output / Side Effects

Path A — Auto-Resolved

Specialist executes domain action (password reset / VPN guide / KB article delivery), closes ticket, marks auto_resolved=true. Coordinator writes full reasoning chain to ticket record via update_ticket. JSON result returned to CLI caller with auto_resolved=true.

Path B — Routed to Queue

Coordinator calls route_ticket assigning to tier1/tier2/networking/security/hardware with calculated SLA timestamp (min of priority SLA and queue SLA). Optional acknowledgement notification sent to user. Audit log written. JSON result with queue and escalated flag returned.

Path C — Escalated

Triggered by: P1 priority, low confidence, security_incident, frozen/terminated account, VIP+P2, schema validation exhausted after 3 retries, or specialist hard stop. Routes to tier2 (or security queue). Full reasoning chain with retry_count and error_types preserved in audit log.

Governance: Every decision path writes a structured reasoning chain to the ticket store via update_ticket. The chain is replayable — given only the log entry a human reviewer can reconstruct exactly what the agent saw and why it decided what it did. Retry count and error types are included in every entry. Auto-resolve scope is defined by the Mandate; any expansion requires a new mandate version and sign-off.
Never automated (by mandate): bulk account operations, hardware replacements, security policy changes, onboarding/offboarding, production database actions, any category where precision falls below 95% in the eval harness.
