Architecture

How it works

Five steps from raw ticket text to resolution or escalation.

Ticket arrives from any channel

Jira, Slack, email, or web portal. The entry point is run_coordinator(ticket) in src/agents/coordinator.py. The coordinator receives the raw ticket body, user ID, and channel — nothing else.

Jira → Slack → Email → run_coordinator()
# src/main.py CLI invocation run_coordinator({ "ticket_id": "T-001", "body": "Can't log in — account locked", "user_id": "U-003", "channel": "portal" })

Coordinator classifies

The coordinator calls classify_ticket via the standard Anthropic messages API with tool use. The model returns a structured result: priority, category, confidence, reasoning, auto_resolvable.

# Expected tool result shape { "priority": "P3", "category": "password_reset", "confidence": 0.92, "reasoning": "Account locked after failed login attempts — standard reset flow", "auto_resolvable": true }

Validation-retry loop

Every classify_ticket result is validated against the TicketClassification Pydantic v2 schema before any downstream decision. On ValidationError, the exact Pydantic error string — which field, which constraint, what value was received — is fed back to the model. Up to MAX_RETRIES = 3 attempts. After exhaustion, escalate with schema_validation_exhausted.

classify_ticket() → Pydantic validate → ValidationError? → retry with error text
# Feedback on retry 2/3 "classify_ticket result failed schema validation (attempt 2/3): 1 validation error for TicketClassification confidence Input should be <= 1 [type=less_than_equal, input_value=1.2, input_type=float] Correct and retry."

Hard-stop gate runs before every tool call

check_pre_tool_use(tool_name, tool_input) is a synchronous Python function — no LLM call, no I/O. It runs before the SDK confirms any tool call on both the coordinator and specialist. It cannot be bypassed by prompt injection because it runs outside the model's control flow.

def check_pre_tool_use(tool_name, tool_input) -> tuple[bool, str]: # 1. payroll_* / finance_* — forward-compat guard # 2. reset_password on frozen/terminated account # 3. SSN pattern in non-PII fields # 4. prompt-injection patterns in route_ticket body return True, "" # or False, deny_message

Specialist resolves end-to-end

If confidence ≥ 0.70, category = password_reset, priority not P1, and account not frozen, the coordinator spawns run_password_reset_specialist() with exactly three fields. The specialist runs its own independent loop and cannot access anything the coordinator knows.

lookup_account → verify_identity → reset_password → close_ticket
# Context isolation: 3 fields only run_password_reset_specialist( ticket_id="T-001", user_id="U-003", issue_summary="Account locked after failed attempts" )
Design

Two agents, one contract

Context isolation is a security property, not a convenience. The coordinator passes exactly three fields.

🧠
Coordinator
src/agents/coordinator.py

Receives every ticket regardless of category. Classifies P1–P4, enriches with user context, decides whether to auto-resolve or route to a human queue. Owns the validation-retry loop. Writes the full reasoning chain to the audit log.

classify_ticket lookup_user route_ticket update_ticket
⚠ No write tools. A prompt-injection attack on the coordinator cannot reset passwords or modify accounts.
🔑
Password Reset Specialist
src/agents/specialists/password_reset.py

Activated only for password resets that pass the auto-resolve threshold. Runs an independent loop with no access to coordinator state: verify identity, reset password, close ticket. Any isError: true from a tool triggers escalation back to the coordinator.

lookup_account verify_identity reset_password close_ticket
⚠ 4 tools per agent — within the reliability sweet spot. Tool-selection degrades past 4–5.
Context Isolation Contract
{

What the specialist cannot see The coordinator's confidence score, retry history, the user's VIP status, the full classification reasoning, or anything else the coordinator knows. If it needs more context, it must acquire it through its own tools. This is a deliberate constraint, not an oversight.
Capabilities

8 tools across 2 agents

Stubs, not live integrations — but every integration point is clearly marked and replaceable.

classify_ticket Coordinator

Returns a TicketClassification struct: priority, category, confidence, reasoning, auto_resolvable. The stub uses keyword signals; production calls the LLM with a structured-output prompt.

body: str channel: str user_id: str
lookup_user Read

Returns directory entry: email, display_name, department, manager_id, account_status, vip_flag. Does not access payroll or financial data.

user_id: str → email, status, vip
route_ticket Write

Assigns ticket to a queue and sets SLA. Does not close or resolve tickets — assignment only. Returns estimated_sla_hours from the config constant.

queue: str priority: str notify: bool
update_ticket Write

Mandatory audit log. Writes the full reasoning chain, routing decision, and retry metadata. Every exit path calls this — routing, auto-resolve, or escalation.

reasoning_chain: str retry_count: int error_types: list
lookup_account Specialist

Returns account status, last_login, failed_attempts, locked_since. Returns ACCOUNT_FROZEN error if status is frozen or terminated — pre-empting the hard-stop check.

user_id: str → status, failed_attempts
verify_identity Read

Challenge/response: employee ID + security answer. No partial match bypass. Security answers are not written to logs. Stub credentials for U-003: EMP-003 / sunshine.

employee_id: str security_answer: str → verified: bool
reset_password Write

Sends temporary credential via email or SMS. Errors with IDENTITY_NOT_VERIFIED if verify_identity was not called first. Errors with ACCOUNT_NOT_ELIGIBLE if account is frozen.

delivery_method: email|sms → temp_credential_sent
close_ticket Write

Marks ticket Resolved with a resolution summary. Called only after reset_password confirms success. Sets auto_resolved=true so metrics can track automation rate.

resolution_summary: str auto_resolved: bool
Safety model

Two layers, different jobs

The hard-stop hook covers categorical prohibitions. The escalation layer covers judgment calls. Neither is asked to do the other's job.

🛑 Hard stops — pre_tool_use.py

🔒
Frozen / terminated account reset reset_password where account_status ∈ {frozen, terminated}
🚫
SSN in non-PII field pattern: \b\d{3}-\d{2}-\d{4}\b outside designated fields
💰
Payroll / financial endpoint tool_name.startswith("payroll_", "finance_")
💉
Prompt injection in routing "ignore prior instructions", "route to CEO" in ticket body

⚠ Escalation rules — coordinator reasoning

priority = P1 → human must acknowledge before any auto-resolve
confidence < 0.70 → escalate with low_confidence reason code
category = security_incident → always escalate regardless of confidence
account_status = frozen → route to security queue
vip + P2 → tier2 with manager notification
retry_count = MAX_RETRIES → schema_validation_exhausted escalation

Layer 1: Hard Stop

Pure Python — no LLM call
Deterministic: same input → same output
Runs outside model control flow
Cannot be bypassed by prompt injection
Legal can unit-test without API calls
Adding a pattern = one if block

Layer 2: Escalation

LLM-driven — coordinator reasoning
Probabilistic by nature
For judgment calls: "should this go to tier2?"
Named conditions in every escalation log
Appropriate for uncertain decisions
Not a substitute for categorical prohibitions
